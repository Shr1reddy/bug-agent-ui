Payment gateway timeout causes silent transaction failure on checkout

Description

When the payment gateway (Stripe) takes longer than the configured 5s timeout, the system silently discards the response without rolling back the order state. The customer sees an error page, but the order record in the database is left in a PENDING state, and no retry or notification is triggered.

This causes inventory to remain locked, duplicate charges on retry, and a broken customer experience. The root cause appears to be in the PaymentService.processCheckout() method where the timeout exception is swallowed.

// PaymentService.ts — problematic section try { const result = await stripe.charges.create(payload, { timeout: 5000 }); await order.setStatus('CONFIRMED'); } catch (err) { // ⚠ timeout swallowed — order stays PENDING logger.warn('Payment failed', err.message); }
Reproduction Steps
  1. Add items to cart and proceed to checkout with a valid card
  2. Intercept the Stripe API call and introduce a 6-second artificial delay (use Charles Proxy or test environment flag STRIPE_DELAY_MS=6000)
  3. Submit the payment form and observe the error page returned to the user
  4. Query the database: SELECT status FROM orders WHERE id = :id — status is PENDING
  5. Confirm no retry event was published to the message queue
  6. Attempt checkout again — observe potential duplicate charge
